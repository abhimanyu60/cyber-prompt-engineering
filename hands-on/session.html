You ask Copilot to analyse a suspicious file and it responds: "This file hash matches CVE-2024-38112, a known Windows MSHTML vulnerability. Severity: CVSS 9.8 (Critical). Patch available since July 2024." What do you do?
ATrust it — the CVE number and CVSS score look specific and credible.
BVerify the CVE on NVD and confirm the hash actually matches before acting — AI can fabricate convincing CVE IDs.
CAI can't analyse file hashes at all.
CVE-2024-38112 is real — but AI can and does generate plausible-looking CVE IDs that don't exist, or incorrectly attribute a hash to a real CVE. Always cross-reference on NVD or VirusTotal before using in a brief or making a patch decision.
You prompt Copilot to summarise a threat bulletin and it states: "This campaign is attributed with high confidence to APT29 (Cozy Bear), a Russian state-sponsored group known for targeting financial institutions." The bulletin itself says attribution is "suspected but unconfirmed." What's the issue?
ANo issue — APT29 is a well-known group so the attribution is likely correct.
BThe issue is that Copilot should have named a different group.
CAI upgraded "suspected" to "high confidence" — a material misrepresentation that could affect regulatory reporting and leadership decisions.
This is a confidence hallucination — AI strips uncertainty from its source material and presents conclusions more definitively than the evidence supports. For attribution, always compare the AI output directly against the original source. Add the constraint: "If attribution is uncertain in the source, preserve that uncertainty in your output."
You ask Copilot to draft a policy clause and it includes: "Under Article 33(1) of GDPR, organisations must notify the supervisory authority within 72 hours of becoming aware of a personal data breach, unless the breach is unlikely to result in a risk to individuals." How do you treat this?
AUse it as-is — GDPR Article 33 is well-established and unlikely to be wrong.
BThe substance is likely correct here, but any regulatory clause in a formal document must be verified against the official GDPR text and confirmed with Legal before use.
CReject it entirely — AI cannot reference legal documents accurately.
In this case, the AI is actually correct — Article 33(1) does say 72 hours. But AI can subtly paraphrase clauses, omit conditions, or cite incorrect article numbers with the same confident tone. Treat AI regulatory output as a helpful starting point — not a verified legal reference. Verify with Legal before including in any formal document.
You ask Copilot to write a detection rule for your SIEM and it produces: "Alert when EventID 4732 fires more than 3 times in 10 minutes from the same source — this indicates a password spray attack." What's the risk?
ANone — EventID 4732 is a Windows security event and this looks like standard SIEM logic.
BEventID 4732 is actually "A member was added to a security-enabled local group" — not a login failure. The rule would never fire for password spray. Verify event IDs before implementing any detection logic.
CThe logic threshold is wrong — it should be 5 attempts, not 3.
The correct EventID for failed logons is 4625 (or 4771 for Kerberos pre-auth failures). EventID 4732 is completely unrelated. AI mixes up specific technical identifiers like EventIDs, port numbers, registry keys, and API names with convincing confidence. Always verify any specific technical identifier before implementing in production systems.
You ask Copilot: "Summarise the key findings from the ISAC report attached." The report is 40 pages. Copilot returns a confident 5-bullet summary. What's the risk?
ANone — summarisation is exactly what AI is designed for and is reliable on long documents.
BAI may hallucinate details not in the report, drop critical caveats, or misrepresent nuanced findings. Spot-check 2–3 bullets directly against the source.
CThe risk is only that it may miss some findings — the findings it does include will be accurate.
AI summarisation is generally strong — but it can silently drop important caveats ("this was only observed in one environment"), merge findings incorrectly, or add plausible-sounding context that wasn't in the source. For high-stakes intelligence, always spot-check a few specific claims against the original before distributing the summary.
To reduce hallucinations in a threat analysis prompt, which constraint is most effective?
A"Be as detailed as possible and include all relevant context you know."
B"Only respond with information you are 100% certain about."
C"Only use information from the data I have provided. If something is uncertain or not in the source data, state that explicitly rather than inferring."
Option C is the gold standard. It grounds the AI in your provided data and explicitly instructs it to surface uncertainty rather than fill gaps confidently. Option A encourages the AI to draw on training data (which may be outdated or wrong). Option B sounds right but AI cannot reliably self-assess its own certainty — it needs the boundary defined by what data it's allowed to use.

Prompt Engineering Cheat Sheet
Reference cards, copy-paste templates, and the constraint toolkit. Keep this tab open during your work sessions.
RCTF Master Template — Copy & Adapt
[ROLE] You are a [specific security expert role]. [CONTEXT] [Describe the situation: system, timeframe, data source, environment]. [TASK] [Exact action required — be specific about the decision or output]. [FORMAT] [Output structure: table / paragraph / JSON / bullets / max length]. [CONSTRAINTS — include as needed] - Only use data provided — do not infer if information is missing, state that explicitly - [Scope constraint: system or timeframe boundary] - [Tone constraint: executive / technical / legal] - [Authority constraint: Tier-1 only / flag and escalate]
🔒 Constraint Toolkit
AccuracyIf data is insufficient, state that — do not infer
ScopeOnly use data from [source / timeframe]
ToneWrite for a non-technical executive audience
AuthorityRecommend only within Tier-1 analyst scope
LegalFlag patterns only — do not conclude on intent
AttributionPreserve uncertainty from the source — do not upgrade confidence
LengthMaximum 200 words / 5 bullets
Hallucin.Do not cite CVEs, EventIDs, or clauses not in the provided data
🧠 Chain-of-Thought Triggers
→"Walk me through your reasoning step by step before concluding."
→"Think through each event in sequence before giving a final assessment."
→"For each finding, show what evidence supports it and what would contradict it."
→"What would you need to see to change your conclusion?"
Use these for: attack path reconstruction, root cause analysis, risk quantification, any analysis where the reasoning must be auditable.
